# 任务11：系统故障恢复（WAL + Redo/Undo日志）

## 提交记录
`4520924` - task11: recovery

## 问题分析
框架提供了日志系统的骨架（LogRecord、LogManager、RecoveryManager），但核心函数均为TODO桩代码。需要实现：
1. 完整的日志记录类型（Commit/Abort/Delete/Update）
2. WAL（Write-Ahead Logging）机制
3. 日志写入缓冲区+刷盘
4. 恢复算法（Analyze → Redo → Undo）
5. 文件头（FileHdr）同步刷盘（关键bug）

## 实现思路

### 1. 日志记录类型
- **BeginLogRecord**: 事务开始（框架已有）
- **CommitLogRecord**: 仅含header（LOG_HEADER_SIZE=20字节）
- **AbortLogRecord**: 仅含header
- **InsertLogRecord**: header + 记录数据 + Rid + 表名（框架已有）
- **DeleteLogRecord**: header + 旧记录数据 + Rid + 表名（新实现）
- **UpdateLogRecord**: header + 旧记录数据 + 新记录数据 + Rid + 表名（新实现）

序列化格式：
```
[LogHeader(20B)] [data_size(4B)] [record_data] [Rid(8B)] [table_name_len(8B)] [table_name]
```

### 2. WAL机制
- **add_log_to_buffer**: `global_lsn++`分配LSN号 → 若buffer满则flush → `serialize`到buffer
- **flush_log_to_disk**: buffer内容写入LOG_FILE_NAME → `persist_lsn`更新 → 清空buffer
- **commit时**: 先写commit日志→flush_log→写文件头→flush_all_pages（保证WAL：日志先于数据）

### 3. 恢复算法
```
analyze(): 从LOG_FILE_NAME读取全部日志到buffer
redo():   遍历日志，对INSERT/DELETE/UPDATE重放操作（将数据页恢复到crash前状态）
undo():   先遍历收集committed事务ID集合，再次遍历，undo未提交事务的操作
          - INSERT→删除记录
          - DELETE→重新插入旧记录
          - UPDATE→恢复旧记录
```

### 4. 关键Bug修复：文件头同步
**问题**: `close_file()`才写文件头到磁盘，crash时`num_pages`等元数据丢失→数据页不可见
**修复**: `commit()`中调用`get_file_hdr()`获取元数据副本→`write_page(fd, RM_FILE_HDR_PAGE)`写入磁盘
**结果**: crash后重启，`open_db`读取到正确的`num_pages`→数据页可见→恢复成功
